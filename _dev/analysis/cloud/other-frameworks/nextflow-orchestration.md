# Nextflow Orchestration Model: Cross-Backend Compute

## Context

Nextflow is a workflow framework (Groovy/JVM) built for bioinformatics at scale.
It has the most mature multi-backend execution story of any workflow tool, with 21
supported executors spanning local, HPC, and cloud. This analysis covers how it
abstracts backend differences and what Artisan can learn from it.

---

## Executor Model

Nextflow's backend abstraction is built on three components:

- **Executor** -- base class that owns a `TaskMonitor` and creates `TaskHandler`
  instances for each task. Subclasses: `LocalExecutor`, `AbstractGridExecutor`
  (parent of SLURM, PBS, LSF, SGE, OAR, Moab, etc.), plus dedicated cloud
  executors for AWS Batch, Azure Batch, Google Cloud Batch, and Kubernetes.
- **TaskHandler** -- per-task lifecycle manager. `GridTaskHandler` handles
  HPC submit/poll via shell commands (`sbatch`, `qsub`). Cloud handlers use
  native SDKs (Java AWS Batch API, GCP client libraries). Each handler manages
  submit, poll status, and collect outputs.
- **TaskMonitor** -- most executors share `TaskPollingMonitor`, which has
  configurable `capacity` (max concurrent submissions), `submitRateLimit`, and
  `pollIntervalMillis`. The monitor owns the submit-poll loop.

Configuration is declarative in `nextflow.config`:

```groovy
process.executor = 'slurm'           // global default
process.queue    = 'compute'

// or per-process via labels:
process {
    withLabel: gpu_tasks { executor = 'awsbatch'; queue = 'gpu-queue' }
    withLabel: quick      { executor = 'local' }
}
```

The key design: **pipeline scripts are executor-agnostic**. Process definitions
declare resource needs; `nextflow.config` maps those to a specific executor. You
switch from SLURM to AWS Batch by changing one config line. Per-process overrides
(via `withName` or `withLabel` selectors) enable hybrid pipelines where different
steps run on different backends.

There are 21 executors total: local, SLURM, PBS/Torque, PBS Pro, LSF, SGE, OAR,
Moab, NQSII, HTCondor, HyperQueue, Flux, Bridge, TCS, Kubernetes, AWS Batch,
Azure Batch, Google Cloud Batch, and the newer Seqera cloud executor (preview).

## Data Transfer & Storage

Nextflow's data model centers on the **work directory** -- a per-task directory
identified by a two-level hex hash (e.g., `work/ab/cd1234...`). Every task gets
an isolated directory containing its staged inputs, wrapper script, stdout/stderr
logs, and outputs.

**Shared filesystem (HPC):** When the work directory is on a shared POSIX
filesystem (NFS, Lustre, GPFS), input files are symlinked into the task directory.
No data copying occurs. This is the simplest mode.

**Object storage (cloud):** The work directory can be an S3 or GCS bucket
(`-work-dir s3://bucket/scratch`). In this mode:

- Nextflow copies input files into the task's S3 work directory before execution.
- The task wrapper script uses the AWS CLI (or equivalent) to stage files from S3
  into the container's local filesystem, run the command, then upload outputs back
  to S3.
- Remote files are staged once at `stage-<session-id>/<hash>/<filename>` and
  reused across tasks that need the same input.

**Automatic transfer:** When input files live on a different filesystem than the
work directory, Nextflow automatically copies them using the appropriate Java SDK.
This makes hybrid workflows (local inputs + cloud execution) transparent.

## Container & Code Delivery

Every Nextflow process runs as a shell script (the `script:` block). Nextflow
wraps this into a `command.sh` that handles staging, execution, and output
collection. The wrapper is generated by `BashWrapperBuilder`.

Container support is a first-class directive:

```groovy
process align {
    container 'biocontainers/bwa:0.7.17'
    // ...
}
```

Multiple container runtimes are supported: Docker, Singularity/Apptainer, Podman,
Charliecloud, and Sarus. Cloud executors (AWS Batch, Google Cloud Batch, K8s) are
**container-native** -- every task runs in a container. HPC executors can
optionally use Singularity.

**Wave containers** (Seqera service) solve container build/delivery:
- Automatically builds containers from Dockerfiles, Conda envs, or Spack packages
  at pipeline launch time
- Pushes to a registry (default: AWS ECR, or user-specified)
- Supports building Singularity-native images on-demand
- Eliminates the "build and push containers before running the pipeline" friction

The critical design choice: **process code is a shell script, not importable
Python**. The container image provides the runtime; the script block provides the
commands. This sidesteps the "worker code must be importable" problem entirely.

## Resource Configuration

Resources are declared as process directives:

```groovy
process heavy_compute {
    cpus 16
    memory '64 GB'
    time '4h'
    disk '100 GB'
    accelerator 4, type: 'nvidia-tesla-v100'    // GPUs
    queue 'gpu-queue'
    clusterOptions '--gres=gpu:4'               // escape hatch for HPC-specific flags
}
```

Key directives: `cpus`, `memory`, `time`, `disk`, `accelerator` (GPUs),
`queue`, `machineType` (cloud VM type). These are portable across executors with
caveats -- `accelerator` is only supported by AWS Batch, Google Cloud Batch, and
Kubernetes. The `clusterOptions` directive is an escape hatch for
backend-specific flags that Nextflow doesn't abstract (e.g., SLURM's
`--constraint`, `--gres`).

Resources can be dynamic, using closures that reference `task.attempt`:

```groovy
memory { 8.GB * task.attempt }    // doubles on retry
```

For AWS Batch specifically, Nextflow creates job definitions with placeholder
resources (1 CPU, 1 GB) and overrides the actual requirements at submission time,
allowing job definition reuse across tasks with different resource profiles.

## Caching & Resume

Nextflow computes a 128-bit hash per task from: session ID, task name,
container/environment image, task script content, all input values and files
(path + size + mtime), global variables referenced in the script, and process
`ext` properties. The hash determines the task's work directory path.

On `-resume`, Nextflow checks whether the cache contains a matching hash AND the
task outputs still exist in the work directory. Both must hold for the task to be
skipped. If any input, script text, or container changes, the hash changes and
the task re-runs.

**Filesystem sensitivity:** By default, file hashing uses path + mtime + size.
NFS environments with inconsistent timestamps can invalidate caches spuriously.
The `lenient` caching mode uses only path + size, trading strictness for
stability on shared filesystems.

**Cloud caching:** When using cloud work directories, the cache database
(`.nextflow/cache/`) is stored as a subfolder of the work directory in S3/GCS,
making it accessible across runs from different machines.

**Cross-backend portability:** The docs do not explicitly claim caching works
when switching executors (e.g., SLURM to AWS Batch). Since the hash includes the
container image and work directory paths change between backends, cache hits
across backends are unlikely in practice. Caching is primarily useful for
**resuming on the same backend** after partial failures or parameter changes.

## Error Handling

The `errorStrategy` directive controls per-process failure behavior:

- `terminate` (default) -- fail the pipeline immediately
- `finish` -- let running tasks complete, then fail
- `ignore` -- mark the task as failed but continue the pipeline
- `retry` -- re-submit the task (up to `maxRetries` times)

`maxErrors` caps the total number of failures across all tasks in a process
before the pipeline is halted (set to `-1` for unlimited retries).

Error strategies can be **dynamic closures** that inspect `task.exitStatus` and
`task.attempt`, enabling patterns like:

- Retry on transient cloud errors (specific exit codes) but fail on logic errors
- Exponential backoff: `sleep(Math.pow(2, task.attempt) * 200)`
- Escalate resources on retry: `memory { 8.GB * task.attempt }`

Failed tasks are retried **independently** -- other tasks in the same process
continue executing. On retry, a new work directory is created (new hash
including attempt number), so partial outputs from the failed attempt don't
interfere.

## Hybrid Execution

A single Nextflow pipeline can use different executors for different processes.
The `withLabel` and `withName` selectors in `nextflow.config` assign executors
per process:

```groovy
process {
    withLabel: local_tasks  { executor = 'local' }
    withLabel: hpc_tasks    { executor = 'slurm'; queue = 'compute' }
    withLabel: cloud_tasks  { executor = 'awsbatch'; queue = 'my-batch-queue' }
}
```

This enables real-world patterns like:
- Lightweight preprocessing on the local machine, heavy compute on SLURM
- Most steps on HPC, GPU steps on AWS Batch
- Quick tasks local, CPU-intensive tasks on a cloud scheduler

When mixing local and cloud executors, separate work directories are needed:
`-work-dir` for local/HPC tasks, `-bucket-dir` for cloud tasks. Nextflow handles
data transfer between them automatically.

The Fusion filesystem further enables hybrid execution by giving POSIX access to
object storage. With Fusion enabled, cloud tasks can read/write S3 paths as if
they were local files, eliminating explicit staging overhead. Fusion uses a lazy
download/upload algorithm that transfers files in the background, and requires
Wave containers + a Seqera Platform license.

## Learnings for Artisan

**Executor = TaskHandler + TaskMonitor pattern.** Nextflow's three-layer
abstraction (Executor owns a TaskMonitor, creates TaskHandlers per task) maps
well to Artisan's planned `BackendBase` refactor. The key insight is separating
the *submission mechanism* (TaskHandler: how to submit/poll one unit of work)
from the *capacity management* (TaskMonitor: how many to submit, how fast, how
often to poll). Artisan's `BackendBase` currently conflates these. Splitting
them would make adding new backends easier -- a Kubernetes backend only needs to
implement submit/poll/collect, not rate limiting or polling loops.

**Work directory as the unit of isolation.** Every Nextflow task gets a
self-contained directory with staged inputs, scripts, and outputs. This is the
mechanism that makes backend portability work -- the directory structure is
identical whether it's on NFS or S3. Artisan's staging model already does
something similar (workers stage Parquet files), but the paths are POSIX-coupled.
Adopting an abstract "task directory" concept that can be backed by either a local
path or an object-store prefix would be the minimal change needed for cloud
support.

**Shell scripts, not importable code.** Nextflow avoids the "worker code must be
importable" problem by having each task be a self-contained shell script running
inside a container. Artisan's Python-import model is deeper and harder to port.
The practical path is to ship a container image with the Artisan package and
user operations pre-installed, rather than trying to transfer code at runtime.
This is what most Nextflow users do -- bake everything into the container.

**Fusion solves the POSIX-to-object-storage gap.** Nextflow hit the exact same
`pathlib.Path` everywhere problem and built a FUSE-based virtual filesystem to
solve it. Artisan doesn't need to build a filesystem, but the lesson is that
**abstracting storage behind an interface** (rather than converting all paths
to URIs) is a viable approach. A thin `StoragePath` abstraction wrapping either
`pathlib.Path` or `s3fs`/`gcsfs` paths, with the same `read_parquet()`/
`write_parquet()` surface, would let existing code work on both local and cloud
storage without rewriting path handling everywhere.

**Caching doesn't port across backends.** Nextflow's cache is effectively
backend-local because the hash includes paths and the work directory structure
differs between backends. Artisan's content-addressed hashing (by artifact
content, not path) is actually better positioned for cross-backend caching.
This is a genuine advantage worth preserving -- if Artisan caches by content
hash rather than file path, cache hits can work regardless of where the artifact
was originally produced.

**Dynamic resource escalation on retry.** Nextflow's pattern of
`memory { 8.GB * task.attempt }` is worth stealing. When a task OOMs on a cloud
backend, automatically retrying with doubled memory is much cheaper than
pre-allocating maximum resources. Artisan's `ExecutionConfig` could support a
similar pattern -- a retry policy that escalates resources per attempt.

**Hybrid execution needs separate work directories.** When mixing backends,
Nextflow requires explicit `-work-dir` and `-bucket-dir` separation. Artisan
should plan for this from the start: the Delta Lake store might live on S3
(accessible from everywhere), but worker scratch space is backend-local. The
orchestrator needs to know which storage is accessible from which backend.
